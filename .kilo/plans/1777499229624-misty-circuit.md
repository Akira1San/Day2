# Task: Refactor models.py into Multiple Modules

## Status

üü° **PLANNED** ‚Äî Not yet implemented

## Problem

`models.py` has grown to ~1887 lines, containing:
- Data classes (Tag, MultiSeriesTag, ScheduleEntry)
- TagManager
- 10+ Strategy classes
- ScheduleGenerator with extensive helper methods

This single-file architecture hinders maintainability, testing, and onboarding.

## Goal

Split `models.py` into focused modules while keeping existing import paths working (zero breaking changes).

## Proposed Module Structure

```
daypart_scheduler/
‚îú‚îÄ‚îÄ data_models.py       # Tag, MultiSeriesTag, ScheduleEntry, TagManager
‚îú‚îÄ‚îÄ strategies.py        # All approximate scheduling strategies
‚îú‚îÄ‚îÄ scheduler.py         # ScheduleGenerator (core engine)
‚îî‚îÄ‚îÄ models.py            # Backward-compatibility shim (re-exports everything)
```

### Module Responsibilities

- **data_models.py** ‚Äî Pure data structures and tag management. No scheduling algorithms.
- **strategies.py** ‚Äî All `*ApproximateStrategy` classes and `CustomTagMergeStrategy`. No ScheduleGenerator logic.
- **scheduler.py** ‚Äî `ScheduleGenerator` class with placement helpers, consume logic, and dispatcher.
- **models.py** ‚Äî Single-line re-exports: `from data_models import *; from scheduler import ScheduleGenerator; from strategies import *`

## Dependency Graph

```
data_models.py  (no internal deps)
    ‚Üë
    ‚îú‚îÄ‚îÄ‚îÄ‚îÄ strategies.py  (imports ScheduleEntry from data_models)
    ‚Üì
scheduler.py  (imports Tag, TagManager from data_models; imports strategies)
    ‚Üì
models.py (re-exports)   ‚Üê  existing imports from all other project files
```

No circular dependencies.
