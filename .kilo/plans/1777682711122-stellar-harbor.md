# Plan: Fix collection_id Assignment in Saved Schedule

## Problem Statement
When saving a schedule, all video entries are getting the same `collection_id` (taken from the tag's collection file's first collection) instead of each video getting its own correct `collection_id` based on which collection it actually belongs to.

## Root Cause Analysis
In `daypart_scheduler.py` `save_schedule()` method, the function `get_schedule_entries_for_day()`:

1. Iterates through all tags to find which tag a schedule entry belongs to (by matching video names)
2. For a matching tag, it calls `get_collection_info(collection_path)` which returns info from the **first collection only** in that collection file
3. It then assigns that single `coll_id` to the `video_info['collection_id']` for ALL videos from that tag

This is wrong because a collection JSON file can contain multiple collections, each with a different `id`. Videos from different collections within the same file should have different `collection_id` values.

## Solution
Modify the code so that each video gets its correct `collection_id` from its source collection.

### Approach A: Enrich videos with collection_id when loading (Preferred)
Modify `utils.load_collection_json()` and `utils.load_collection_videos_only()` to add the collection's `id` to each video dict as `collection_id`. Then the schedule save code can simply use `video.get('collection_id', '')` directly.

**Pros:**
- Clean fix at the data loading layer
- Collection ID travels with video everywhere
- Minimal changes to save_schedule logic
- No performance impact

**Cons:**
- Need to ensure all loading code paths are updated

### Approach B: Fix in get_schedule_entries_for_day matching
Create a more sophisticated matching that looks up the specific collection ID for each video path from the collection file.

**Pros:**
- Doesn't modify data loading

**Cons:**
- More complex
- Performance overhead (repeated file lookups or caching per video path)
- Messier code

## Recommended Implementation (Approach A)

### Changes Required:

1. **utils.py** - Modify `load_collection_json()`:
   - When iterating through collections, for each video in a collection, add `collection_id` (and optionally `collection_name`, `channel`) to the video dict before appending to collection_videos.

2. **utils.py** - Modify `load_collection_videos_only()`:
   - Similarly add `collection_id` to each video dict.
   - Also ensure the function returns videos with their collection metadata when multiple collections exist.

3. **daypart_scheduler.py** - Modify `save_schedule()`'s `get_schedule_entries_for_day()`:
   - After finding matching tag and video, directly use `video.get('collection_id', '')` from the matched video dict instead of using `coll_id` from tag's collection info.
   - The fallback case (line 558) can remain as-is or use empty string.

4. **Testing**:
   - Run existing tests to ensure nothing breaks
   - Manually verify that schedule JSON now contains correct per-video collection_id values

## Edge Cases
- Videos that don't belong to any collection (should have empty collection_id)
- Legacy tags or videos loaded from old format
- Multiple collections in one file with different IDs
- Videos that appear in multiple collections (should use whichever collection they matched from in the tag)
