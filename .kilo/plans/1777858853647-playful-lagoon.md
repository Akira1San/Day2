# Code Review: dialogs.py

## Critical Issues

### 1. Missing `self.tag` attribute in RandomFillDialog
**Location**: `RandomFillDialog.__init__` (line 554)

The `tag` parameter is passed but never stored as `self.tag`. Later, `load_channels()` (line 1052) references `self.tag` which will cause `AttributeError`.

**Fix**: Add `self.tag = tag` in `__init__` after the super call.

---

### 2. Inconsistent blacklist handling in TagDialog.load_collection
**Location**: TagDialog.load_collection (lines 357-399)

The blacklist loading logic (lines 370-382) loads blacklist_data but conditionally assigns it to `self.blacklist` only inside the `if load_blacklist` block (line 382). However, the subsequent loop that adds videos to blacklist (lines 920-925) assumes `blacklist_data` exists but it may be empty if `load_blacklist` was False. This is also duplicated in RandomFillDialog.load_collection (lines 881-944).

**Fix**: Consolidate blacklist loading logic; ensure `self.blacklist` is always defined before use. The current code also has a bug where videos from the blacklist file that aren't in collection_videos get added anyway (lines 923-925).

---

### 3. Unused import
**Location**: Line 14

`from PySide6.QtGui import QFont` is imported but never used.

**Fix**: Remove the import.

---

### 4. Potential duplicate video in blacklist (TagDialog.add_to_blacklist)
**Location**: TagDialog.add_to_blacklist (lines 308-318)

The check `if v not in self.blacklist` compares dictionary objects, but since `v.copy()` is used when adding to `self.collection_videos` (line 295), the comparison might fail due to different object identities even if the paths are equal.

**Fix**: Use path-based comparison: `if not any(b.get('path') == v.get('path') for b in self.blacklist)`.

The same issue exists in RandomFillDialog.add_to_blacklist (lines 985-994).

---

### 5. RandomFillDialog.get_tag: profile names vs paths
**Location**: RandomFillDialog.get_tag (lines 1079-1103)

`collection_profile` and `blacklist_profile` are set from `combo.currentText()` which contains filenames, not full paths. This is inconsistent with how TagDialog.get_tag handles it (lines 533-539) and what the Tag model expects. The Tag model fields `collection_profile` and `blacklist_profile` are meant to be profile names (filenames), but some parts of the code may treat them as paths.

**Review**: Verify the intended semantics of these fields. If they should be filenames, this is correct but inconsistent with the `blacklist_path` field usage. If they should be paths, the combo boxes should store full paths in item data (like `load_available_collection_profiles` does for blacklist profiles already).

---

### 6. RandomFillDialog.save_blacklist_file uses wrong file extension
**Location**: RandomFillDialog.save_blacklist_file (lines 1039-1050)

It saves to `_blacklist.json` but `load_blacklist_file` accepts both `.json` and `.ini` files. The TagDialog.save_blacklist_file (lines 500-510) also uses JSON. However `load_blacklist_json` in utils.py can load both formats. This is not a bug but worth noting the inconsistency in the file naming convention across the app.

---

### 7. RandomFillDialog.load_available_profiles duplicates TagDialog.load_available_collection_profiles
**Location**: Two nearly identical methods (lines 401-442 vs 793-834)

Identical logic with minor naming differences. This is a maintenance hazard.

### 8. SeriesDialog incomplete implementation
**Location**: SeriesDialog (lines 1202-1476)

- Does NOT inherit blacklist functionality from BaseTagDialog properly. It calls `super().__init__` which initializes `self.blacklist` and `self.collection_videos`, but then the UI doesn't include blacklist management at all. The blacklist is used only for filtering in `get_tag` (line 1473) but there's no UI to edit it.
- The `auto_calc_end_time` method (1426-1448) calls `self.start_time_edit` and `self.end_time_edit` which are defined in BaseTagDialog's `_setup_time_inputs`, but these are never called in SeriesDialog's `setup_ui`. Actually, `SeriesDialog.setup_ui` does call `self._setup_time_inputs` at line 1319, so these exist. But BaseTagDialog's `__init__` sets `self.collection_videos = []` and `self.blacklist = []` which SeriesDialog then overwrites.
- `blacklist_profile_selected` (1408-1415) calls `self.load_blacklist_file` but no such method exists in SeriesDialog (it's only in BaseTagDialog). Wait, BaseTagDialog does NOT define `load_blacklist_file`. Actually looking back, `load_blacklist_file` is defined in TagDialog and RandomFillDialog, NOT in BaseTagDialog. So SeriesDialog's call to `self.load_blacklist_file` at line 1415/1424 will crash because that method doesn't exist in any parent class.
- SeriesDialog doesn't implement `_create_video_list_section` or `_create_blacklist_section` like TagDialog/RandomFillDialog do. But it doesn't need them because its UI is simpler.
- SeriesDialog does not have a `_setup_time_inputs` method of its own but calls it; it inherits from BaseTagDialog which has it (line 33-47). So that's okay.

**Summary**: SeriesDialog appears to be a simplified dialog without blacklist UI, but it still tries to call methods like `load_blacklist_file` and manage `self.blacklist` which are not implemented properly. It either needs full blacklist support (like the other two dialogs) or should remove blacklist-related code entirely.

### 9. SeriesDialog.load_blacklist_file missing

**Fix**: Either implement `load_blacklist_file` in SeriesDialog or remove blacklist profile selection support entirely if blacklist editing is not part of SeriesDialog's scope.

### 11. TagDialog.load_collection: blacklist appended incorrectly
**Location**: TagDialog.load_collection lines 923-925

After the main loop that populates `self.blacklist` from videos that match blacklist entries, there's an additional loop:
```python
for bl_video in blacklist_data:
    if bl_video not in self.blacklist:
        self.blacklist.append(bl_video)
```
This adds blacklist entries that were NOT found in the collection (e.g., if the blacklist file contains paths that don't exist in the collection). The intended behavior is unclear. If the goal is to have a complete blacklist regardless, fine. But in RandomFillDialog.load_collection, the equivalent code (lines 923-925) does NOT have this second loop — meaning the two dialogs behave differently.

**Fix**: Decide on consistent behavior. If blacklist should only include videos that exist in the collection, remove the second loop. If it should include all entries from the blacklist file, keep it and add the same to RandomFillDialog.

However, for any blacklist items added elsewhere or the "-- None --" placeholder (index 0), itemData may be None. The code handles that with the `if bl_path` check, so it's okay. Just noting.

## Code Quality & Design Issues

### 1. Massive code duplication between TagDialog and RandomFillDialog
Both classes share:
- `_create_video_list_section` (identical)
- `_create_blacklist_section` (identical except TagDialog's save_blacklist_file uses JSON; RandomFillDialog's also uses JSON but with different message box type)
- `browse_collection`
- `load_available_profiles` / `load_available_collection_profiles` (almost identical)
- `collection_profile_selected` vs `profile_selected` (identical logic)
- `blacklist_profile_selected` (identical)
- `load_blacklist_file` (slightly different: TagDialog sets `self.blacklist_path`, RandomFillDialog also sets `self.blacklist_path`)
- `save_blacklist_file` (almost identical, but RandomFillDialog uses `QMessageBox.warning` instead of `information`)
- `on_video_selected`, `select_all_videos`, `clear_selection`, `add_selected_videos`, `remove_selected_added`, `remove_all_added`, `add_to_blacklist`, `remove_from_blacklist`, `refresh_added_list`, `refresh_blacklist_list`, `update_counts`
- `auto_calc_end_time` (slightly different logic for randomization)

**Recommendation**: Extract a common base class (e.g., `VideoSelectionDialogBase`) that implements all these shared methods. TagDialog and RandomFillDialog would then inherit from it and only override:
- `setup_ui` (for the unique UI layouts)
- `get_tag` (for the specific Tag type returned)
- Any unique behavior (e.g., `auto_calc_end_time` logic, `save_blacklist_file` message type)

---

### 2. Inconsistent copying of video dictionaries
Throughout the code, videos are sometimes stored as `video.copy()` and sometimes just `video` (reference). For example:
- TagDialog.add_selected_videos: `self.added_videos.append(video.copy())` (line 295)
- RandomFillDialog.add_selected_videos: `self.added_videos.append(video)` (line 972) — no copy!
- TagDialog.add_to_blacklist: `self.blacklist.append(v.copy())` (line 314)
- RandomFillDialog.add_to_blacklist: `self.blacklist.append(v)` (line 991) — no copy!

This could cause unintended mutations if the original collection_videos are modified elsewhere.

**Fix**: Be consistent. Prefer `.copy()` to avoid aliasing. Or better, define a proper data class for videos.

---

### 3. Duplicate section creation methods inside same class
`TagDialog._create_video_list_section` and `TagDialog._create_blacklist_section` (lines 189-267) define local classes using `type('VideoSection', (), {...})()` which is unnecessarily cryptic. They return simple objects with `widget`, `videos_list`, `count_label`. This pattern is repeated exactly in `RandomFillDialog._create_video_list_section` (lines 705-751) and `_create_blacklist_section` (lines 753-783).

**Recommendation**: Use a `@dataclass` or simple `namedtuple` for clarity. Or just return a tuple or dict.

---

### 4. Inconsistent blacklist save format
TagDialog.save_blacklist_file (line 504) creates JSON with `{'blacklist': self.blacklist}` structure. utils.load_blacklist_json expects either INI format or JSON possibly containing a 'blacklist' key. This is fine, but document it.

---

### 5. RandomFillDialog.load_available_profiles: inefficient set usage
Lines 806-828 create a dict `blacklist_files`, then convert keys to sorted list. The `set` would be sufficient if we only needed names, but we need paths too. The logic to avoid duplicates is fine. But later at lines 830-834, it iterates sorted(blacklist_files.keys()). Could simplify: since dict keys are unique already, just `for name in blacklist_files:` and rely on dict maintaining insertion order, but sorted is fine for determinism.

Minor: lines 816-828 have 4 similar glob loops that could be combined into a list of patterns and a double loop.

---

### 6. SeriesDialog.load_available_profiles uses a set
Lines 1364-1376 uses `blacklist_files = set()` and only adds names (not paths). This means the `load_blacklist_file` call at line 1405/1415 would have no stored path, so it tries to construct from `blacklist_path`. But SeriesDialog never defines `get_config_paths` import? Actually it's imported at top from utils. So it would work. However, the set discards the file path, so later code cannot retrieve the full path without re-globbing. In TagDialog/RandomFillDialog, they store the full path as itemData so `blacklist_profile_selected` can directly load. SeriesDialog doesn't store itemData, and `blacklist_profile_selected` tries to call `self.load_blacklist_file` which doesn't exist. So the whole blacklist profile combo in SeriesDialog is broken.

---

### 7. BaseTagDialog.__init__ signature mismatch
**Location**: BaseTagDialog (line 27) and SeriesDialog.__init__ (line 1203-1204)

SeriesDialog calls `super().__init__(parent, tag)`. BaseTagDialog's `__init__` signature is `def __init__(self, parent=None, tag: Optional[Tag] = None)`. That's fine. But BaseTagDialog's init only sets `self.collection_videos = []` and `self.blacklist = []`. It does NOT store `self.tag = tag`. TagDialog and RandomFillDialog don't store it either, but they both later reference `self.tag` (TagDialog at line 69 `if tag:` and also at 78 `tag.collection_path`; but those are the local `tag` param, not self.tag). RandomFillDialog at line 565 also uses the `tag` param directly. So no current code uses `self.tag` except `RandomFillDialog.load_channels()` at line 1058 which checks `hasattr(self, 'tag') and self.tag`. Since `self.tag` is never set, that check fails. So load_channels probably doesn't work correctly when editing an existing tag.

**Fix**: Set `self.tag = tag` in BaseTagDialog.__init__ (or in child __init__) and ensure it's available.

### 8. TagDialog and RandomFillDialog: load_collection calls filter_videos_by_blacklist twice
TagDialog.load_collection:
- line 95: after loading tag's added_videos, it calls `self.added_videos = filter_videos_by_blacklist(...)`
- line 396-397: if randomfill config, adds videos to added_videos without blacklist filtering; but then line 399 calls refresh_added_list. However, the filtering may be missing for those auto-added videos? Actually line 393 checks `get_randomfill_config()`, and then lines 394-397 add all non-blacklisted videos. That's fine. But earlier, at line 95, it filters `self.added_videos` after loading from tag. So overall okay, but the logic order is confusing.

RandomFillDialog.load_collection (lines 938-944) also does similar: auto-adds non-blacklisted videos. That's consistent.

### 9. ConfigDialog.save_config: writes empty values
If `self.collection_path_edit.text()` is empty, `config['Paths']['collection_path']` will be empty string, overwriting the fallback. Consider only writing non-empty values, or reading existing config and updating only changed fields.

Not a bug, but behavior may surprise users.

---

### 10. SeriesDialog auto_calc_end_time calls parse_videos_for_series but doesn't store results
Line 1433-1438 calls parse_videos_to_series but ignores the returned list, only sums durations. That's fine for the calculation, but it's wasteful to compute the full parsed list if only durations are needed. However the function also returns an index; could optimize? Not necessary.

### 11. Inconsistent naming: `_create_video_list_section` returns an object with `.videos_list` and `.count_label` but the actual QListWidget variable is named differently in each dialog. That's okay but confusing. In TagDialog it assigns: `collection_widget = self._create_video_list_section(...); self.videos_list = collection_widget.videos_list`. This is fine.

### 12. Potential for infinite loop in profile_selected signal handling
TagDialog.profile_selected (line 444) changes the blacklist_profile_combo currentIndex (line 463), which will trigger `blacklist_profile_selected`. That method may call `load_blacklist_file` which updates UI. This could cascade. However it's likely intentional, but could cause re-entrancy if not careful. Current code seems safe because blacklist_profile_selected only loads if index > 0 and file exists.

## Minor Observations

- `BaseTagDialog._setup_time_inputs`: default times are QTime(0,0) and QTime(1,0) (1 AM). That's an odd default end time; maybe should default to same as start or 1 hour later? Not necessarily a bug.
- `BaseTagDialog._load_collection_to_list`: unused method (defined but never called). Remove.
- Many debug logger statements with `[DEBUG]` prefix (lines 413, 438, 448, etc.). Acceptable but could use logger.debug() directly with structured logging.
- TagDialog line 74: `if hasattr(tag, 'randomize_videos') and tag.randomize_videos:` Since Tag class always has `randomize_videos` attribute (default False), hasattr is unnecessary.
- In RandomFillDialog.__init__ line 570: `fill_24h = getattr(tag, 'fill_24h', False)` — okay because Tag objects may not have fill_24h if they were created before that field existed. Good use of getattr.
- SeriesDialog doesn't use blacklist at all except get_tag includes it. The UI doesn't allow editing blacklist. Should either add UI or remove blacklist field from Tag creation.
