# Plan: Fix Season Sequence Multi-Day Bug

## Problem
Series tags in `season_sequence` mode only play first season episodes, then subsequent days only play episodes 1 & 2 from the next season. This occurs when scheduling multiple days in approximate modes (find_replace, early_fill, late_fill, priority, linear_spanning, exhaustive). Root cause: `_place_tag_videos` in `scheduler.py` does not handle `season_sequence` mode correctly for single-series tags — it uses modulo-wrapping logic instead of the season-aware `_select_series_videos` helper.

## Analysis
- `_select_series_videos` (scheduler.py:44) correctly implements season_sequence using flat ordered list and a global index based on `day_offset * video_count`.
- `_process_series_tag` (scheduler.py:282) correctly calls `_select_series_videos`.
- `_apply_approximate_linear` (scheduler.py:918) correctly calls `_select_series_videos`.
- `_place_tag_videos` (scheduler.py:106) handles single-series tags with hardcoded modulo logic, ignoring season_sequence. This method is used by many approximate strategies (find_replace, early_fill, etc.) and by `_process_multi_series_tag` for each series config inside a MultiSeriesTag. However, for multi-series it already calls `_select_series_videos`; only the single-series branch is broken.

Consequences:
- Season_sequence with approximate strategies wraps around after exhausting episodes or produces incorrect ordering.
- If daily slot fits fewer episodes than `video_count`, episodes are lost because offset jumps regardless.

## Solution
1. Modify `_place_tag_videos` (single-series branch) to use `_select_series_videos` when `play_mode == "season_sequence"` and `_has_season_tags` is true.
2. Keep existing fallback for `sequence` and `random` modes.
3. This unifies episode selection across all code paths.
