# Plan: Fix Time Truncation to Include Seconds

## Problem Statement

The schedule generator outputs times in the format `HH:MM:00` (seconds always zero). When used with streaming videos, the first few seconds of videos are cut off. The old schedule (from a previous version) contained times like `08:42:22` - with non-zero seconds. This indicates that the current implementation incorrectly truncates all times to minute boundaries, causing scheduling misalignment.

## Root Cause

- Internal time representation uses **minutes** (`start_minutes`, `end_minutes`) as integers.
- Video durations (in seconds) are truncated via `// 60` to get whole minutes, discarding remainder seconds.
- Start/end times are computed by adding truncated minute durations, losing accumulated seconds.
- JSON export hardcodes seconds to `"00"`:  
  ```python
  time_str = f"{start_h:02d}:{start_m:02d}:00"
  ```
- This causes each video to start **earlier** than its true playback position (because previous video's actual duration was longer than the truncated minutes), cutting the first seconds of the following video.

## Proposed Solution

Switch the entire scheduler to **second-precision** timing:

1. Change `ScheduleEntry` to store `start_seconds` and `end_seconds` (int).
2. Convert all tag time conversions (`QTime`) to total seconds via new `qtime_to_seconds`.
3. Use full video duration in seconds (no `// 60` truncation).
4. Update all arithmetic, comparisons, loops, and thresholds to operate in seconds.
5. Keep UI preview at minute resolution (floor seconds -> minutes) for display.
6. Output JSON times with correct seconds: `HH:MM:SS`.

This preserves accumulated seconds from video durations, ensuring accurate start times and preventing video cuts.

## Detailed Changes

### 1. `data_models.py`

- **ScheduleEntry**
  - Change `__init__(day, start_minutes, end_minutes, ...)` to `__init__(day, start_seconds, end_seconds, ...)`.
  - Store `self.start_seconds`, `self.end_seconds`.
  - Add backward-compatibility properties:
    ```python
    @property
    def start_minutes(self):
        return self.start_seconds // 60

    @property
    def end_minutes(self):
        return self.end_seconds // 60
    ```
  - Update `to_display_string` and `to_copy_string`:
    - Compute day: `(self.start_seconds // (24 * 3600)) + 1`
    - Compute hours: `(self.start_seconds // 3600) % 24`
    - Compute minutes: `(self.start_seconds % 3600) // 60`
    - Keep display format `HH:MM` (ignore seconds) for UI.
  - `format_time` method updated similarly (if used).

### 2. `utils.py`

- Add:
  ```python
  def qtime_to_seconds(qtime: QTime) -> int:
      return qtime.hour() * 3600 + qtime.minute() * 60 + qtime.second()
  ```
- Keep existing `qtime_to_minutes` for any remaining uses (will be phased out).

### 3. `scheduler.py`

- **Replace minute-based variables with seconds** throughout.
- `_create_video_entry`: use seconds for `pos` and `duration`.
- `_build_random_entries`:
  - Parameters `start_pos`, `end_pos` are seconds.
  - `duration = int(video.get('duration', 90))` (seconds, no `// 60`).
  - Loop with `pos` in seconds.
  - Call `ScheduleEntry` with seconds.
- `_process_custom_tag`, `_process_series_tag`, `_process_multi_series_tag`:
  - Convert tag start/end via `qtime_to_seconds`.
  - `start_offset` and `day_offset` expected in seconds (callers must supply seconds).
  - Loop `for m in range(start_sec, end_sec): occupied.add(m)`.
- `_process_random_fill_tag`:
  - All positions (`gap_start`, `gap_end`, `pos`) in seconds.
  - Duration in seconds.
- `_consume_overlapping_tail`: parameters `slot_start`, `slot_end`, etc. in seconds.
- `_approximate_finalize_day`:
  - `day_end = day_start + 86400`.
  - Replace all `.start_minutes` / `.end_minutes` with `.start_seconds` / `.end_seconds`.
  - Adjust debug logs to format seconds as `HH:MM:SS` (optional but helpful).
- `_apply_approximate_find_replace`:
  - `total_seconds = num_days * 86400`.
  - `day_start = day_offset * 86400`, `day_end = day_start + 86400`.
  - `THRESHOLD_AFTER = 30 * 60` (1800 seconds).
  - Replace all `qtime_to_minutes` with `qtime_to_seconds`.
  - All arithmetic with positions uses seconds.
  - Update log formatting to show seconds.
- `_apply_approximate_linear`:
  - Convert `day_offset_minutes` to `day_offset_seconds = day_offset * 86400`.
  - Use seconds for `original_start`, `original_end`, `custom_start`, `custom_end`, etc.
  - Duration in seconds, no truncation.
  - Occupied marking loops over seconds.
  - `generate_random_fill` calls: pass seconds.
- `generate_random_fill`:
  - Parameter rename `remaining_seconds` (or keep `remaining_minutes` but convert). Prefer rename.
  - `current_second` loop.
  - `duration = int(video.get('duration', 90))`.
  - `end_second = min(current_second + duration, remaining_seconds)`.
- Update any other helper methods.

### 4. `daypart_scheduler.py`

- **Preview generation (weekly/monthly):**
  - Compute from `entry.start_seconds`:
    ```python
    start_h = (entry.start_seconds // 3600) % 24
    start_m = (entry.start_seconds % 3600) // 60
    ```
- **`save_schedule`** – `get_schedule_entries_for_day`:
  - Build `time_str` with seconds:
    ```python
    secs = entry.start_seconds % 86400
    h = secs // 3600
    m = (secs % 3600) // 60
    s = secs % 60
    time_str = f"{h:02d}:{m:02d}:{s:02d}"
    ```
  - Adjust any comparisons using `entry.start_minutes` → `entry.start_seconds`.
- Remove any minute-based conversions from entry.

### 5. `strategies.py`

- **All strategy classes** (CustomTagMergeStrategy, FindReplaceApproximateStrategy, LinearApproximateStrategy, EarlyFillApproximateStrategy, LateFillApproximateStrategy, PriorityApproximateStrategy, LinearSpanningApproximateStrategy, ExhaustiveApproximateStrategy):
  - Replace `qtime_to_minutes` with `qtime_to_seconds`.
  - Replace constants `24*60` with `24*3600`; `1440` with `86400`.
  - Change all `e.start_minutes`, `e.end_minutes` to `e.start_seconds`, `e.end_seconds`.
  - Where durations are computed (`duration = orig_end - orig_start`), ensure they're in seconds.
  - Update any `total_minutes` variables to seconds.
  - In `inject_into_random` (CustomTagMergeStrategy):
    - `start`, `end` in seconds.
    - Loop conditions using seconds.
    - Video duration full seconds.
    - `total_duration` sum of seconds.
    - `avg_duration` in seconds.
  - In `PriorityApproximateStrategy`, change `THRESHOLD_AFTER = 30` → `30 * 60`.
  - Update sorting and comparisons to use seconds.
  - Anywhere `start_minutes` used for sorting or range checks, update.

### 6. `dialogs.py` & other UI files

- No direct time arithmetic; only QTime edits. No changes required.

### 7. Backward-Compatibility

- Provide `start_minutes` and `end_minutes` properties on `ScheduleEntry` (as above) so any external code or tests using the old attribute names continue to work, returning floor(seconds/60).
- Keep `qtime_to_minutes` for any external imports.

### 8. Tests

- Existing test files likely use `.start_minutes` for assertions. Thanks to compatibility properties, they should continue to work without modification.
- However, tests that hardcode expected minute values may need adjustment if they rely on exact durations; but since minute values are derived from `seconds // 60`, they remain identical to previous behavior.
- Recommend running all tests after conversion.

## Constants Conversion Summary

| Meaning                     | Old (minutes) | New (seconds) |
|----------------------------|---------------|---------------|
| 1 hour                     | 60            | 3600          |
| 1 day                      | 1440          | 86400         |
| 24 * 60                    | 1440          | 86400         |
| THRESHOLD_AFTER (30 min)   | 30            | 1800          |
| Random fill default (90)   | 90 sec -> 1.5 min? Actually duration default 90 seconds. No change; we keep raw seconds. |

## Risks & Considerations

- Performance: Marking occupied seconds individually increases set size (max 86400 per day). Still acceptable for typical schedules.
- Memory: `occupied` set could hold up to 86400 * number_of_days entries; still modest (few MB).
- Extensive changes increase chance of missing a spot. Must ensure all minute-based arithmetic is replaced.
- Debug logs: formatting in `_approximate_finalize_day` and elsewhere currently uses `//60%24` and `%60`; update to show seconds properly to avoid misleading logs.
- Ensure no division by 60 assumptions remain except when converting to display minutes.

## Verification Steps

1. Generate a schedule with videos having non-round durations (e.g., 95 sec, 150 sec).
2. Inspect saved JSON: times should show seconds (e.g., `00:01:35`, `00:04:10`).
3. Simulate streaming: verify no video starts before previous one ends (check overlapping).
4. Run existing unit tests to catch regressions.
5. Manual UI check: preview still shows HH:MM correctly.
