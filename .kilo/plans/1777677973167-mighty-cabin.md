# Plan: Fix multi_series Tag Serialization to Store Collection Paths Instead of Video Names

## Problem
Multi-series tags currently save full video lists (including video names, durations, paths) inside the `.ini` file as JSON. This bloats the file size and duplicates data that already exists in collection files. In contrast, series tags, custom tags, and random fill tags store only the path to the collection file, keeping configuration files small and maintainable.

## Objective
Change multi_series tag serialization so that only metadata and collection paths are saved (like series tags). Videos should be loaded from the collection files at runtime.

## Current State
- `SeriesConfigDialog.get_config()` returns a dict containing both `collection_videos` (full list) and `collection_path`.
- `MultiSeriesTag` stores a `series_list` where each entry has `collection_videos`.
- `serialize_tag_to_string()` dumps the entire `series_list` as JSON, including the large `collection_videos` arrays.
- `deserialize_tag_from_string()` loads the series_list as-is, preserving `collection_videos`.

## Proposed Changes

### 1. Strip `collection_videos` during serialization (serialization.py)
In `serialize_tag_to_string()`, under the `is_multi_series` branch:
- Create a cleaned version of `series_list` that excludes the `collection_videos` key from each entry.
- Dump only the cleaned list.

```python
if getattr(tag, 'is_multi_series', False):
    lines.append(f"type = multi_series")
    lines.append(f"name = {tag.name}")
    lines.append(f"start_time = {tag.start_time.toString('HH:mm')}")
    lines.append(f"end_time = {tag.end_time.toString('HH:mm')}")
    # Remove collection_videos to avoid bloating the ini file
    clean_series_list = [
        {k: v for k, v in s.items() if k != 'collection_videos'}
        for s in tag.series_list
    ]
    lines.append(f"series_list = {json.dumps(clean_series_list)}")
```

### 2. Load videos from paths during deserialization (serialization.py)
In `deserialize_tag_from_string()`, under the `multi_series` branch:
- After `series_list = json.loads(series_list_str)`, iterate over each series entry.
- For each entry, if `collection_path` is provided, load videos via `load_collection_videos_only(collection_path)` and set `series['collection_videos']`.
- If the path is missing or loading fails, set `collection_videos` to an empty list.
- This ensures in-memory objects always have `collection_videos` for scheduling logic, regardless of whether the data came from an old file (with embedded videos) or a new file (with only a path).

```python
elif tag_type == 'multi_series':
    series_list_str = tag_section.get('series_list', '[]')
    try:
        series_list = json.loads(series_list_str)
    except:
        series_list = []
    # Populate collection_videos from collection_path for each series
    for series in series_list:
        coll_path = series.get('collection_path', '')
        if coll_path:
            try:
                series['collection_videos'] = load_collection_videos_only(coll_path)
            except Exception:
                series['collection_videos'] = []
        else:
            series['collection_videos'] = []
    from models import MultiSeriesTag
    return MultiSeriesTag(
        name=name,
        start_time=qtime_from_string(start, 'HH:mm'),
        end_time=qtime_from_string(end, 'HH:mm'),
        series_list=series_list
    )
```

### 3. No changes required elsewhere
- All processing code (scheduler, MultiSeriesTag methods) already reads `collection_videos` from each series config. Since deserialization now guarantees that key exists (populated from the path), no changes are needed there.
- The UI dialogs already collect and store `collection_path` and `collection_videos`; videos are in memory for preview, and serialization will strip them automatically.
- No changes to `MultiSeriesDialog.get_tag()` or `SeriesConfigDialog.get_config()` are strictly necessary, though optional cleanup could be done later.

## Backward Compatibility
- **Old .ini files** with embedded `collection_videos` will still load because the deserialization code preserves existing data; it will simply overwrite with fresh load from path (if path exists), which is desirable.
- **New .ini files** will contain only paths, reducing file size significantly.

## Testing
1. Open an existing multi_series tag (pre-change) and verify scheduling still works.
2. Save a multi_series tag, then inspect the `.ini` file to confirm `series_list` JSON contains no large video arrays (only paths and metadata).
3. Reload the saved file and run the scheduler to ensure behavior is unchanged.
4. Check that series with missing collection files produce sensible placeholder entries.

## Risks & Mitigations
- **Risk**: A series entry without a valid `collection_path` results in empty videos, breaking the schedule.
  **Mitigation**: The UI forces selection of a collection file when adding a series, so `collection_path` should always be non-empty. Deserialization gracefully handles missing files by returning an empty list, which triggers a placeholder entry (existing logic already handles that).
- **Risk**: Paths could be relative and break when moved.
  **Mitigation**: Currently all paths are absolute (from QFileDialog). No change needed.
