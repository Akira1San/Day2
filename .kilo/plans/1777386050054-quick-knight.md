## Detailed Steps

### 1. Fix Truncation Bug (`_apply_approximate_find_replace`)

**Status:** NOT-DONE (confirmed in current code at models.py:641-859). The approximate find-replace algorithm loses random-fill segments before custom/series tags in two ways:

- **Missing head portion of overlapping random entries:** When a random-fill entry overlaps the tag's start (i.e., starts before `new_start` and ends after `new_start`), only its tail after the tag is preserved (lines 742-755, 763-776, 787-800). The portion before `new_start` is discarded, creating gaps/truncation before custom tags.
- **Backward current_pos updates in post-tag fill:** The loop at lines 816-840 contains `current_pos = rand_e.end_minutes` in the branch where `rand_e.end_minutes <= current_pos`. This moves `current_pos` backwards, causing later entries to be skipped and distorting the schedule after custom tags.

**Fix:**

- In the overlapping-entry handling blocks (three branches, all with `if re.start_minutes < new_end and re.end_minutes > new_start:`), before placing the tag or after, add the *head* portion:
    ```python
    # Add head portion if re starts before new_start
    if re.start_minutes < new_start:
        final.append(ScheduleEntry(1, re.start_minutes, new_start, re.video_name))
    ```
  Do this before the tag's videos are placed (i.e., before calling `_place_tag_videos` for the current tag, or insert into `final` in the correct time order; final sort will reorder).

- Remove the line `current_pos = rand_e.end_minutes` from the `if rand_e.end_minutes <= current_pos` branch in the post-tag fill loop (line ~830). `current_pos` should never regress; it should only advance when we append entries that end after it.

- In the same loop's `elif` branch where a tail is added (`rand_e.start_minutes < current_pos < rand_e.end_minutes`), `current_pos` is correctly set to `rand_e.end_minutes` (forward). Keep that.

These changes ensure:
  - Overlapping random entries contribute both their pre-tag head (if any) and post-tag tail (if any), with the tag occupying the middle.
  - `current_pos` monotonically increases, preventing skipped slots.

### 11. Refactor `Tag` God Object

**File:** `models.py:11-74`

**Current state:** NOT-STARTED. The `Tag` class remains a single God object with ~20 init parameters; no inheritance hierarchy introduced yet.

**Approach:** Subclass `Tag` into specialized types rather than using `tag_type` + boolean flags:

```python
class Tag:
    def __init__(self, tag_type, name, start_time, end_time, ...): ...

class CustomTag(Tag):
    def __init__(self, name, start_time, end_time, collection_videos, randomize_videos, video_count): ...

class SeriesTag(CustomTag):
    def __init__(self, name, start_time, end_time, collection_videos, start_season, start_episode, play_mode, video_count): ...

class RandomFillTag(Tag):
    def __init__(self, name, start_time, end_time, collection_videos, blacklist, fill_24h, collection_profile): ...
```

**Note:** This is a large refactor — the `Tag` class is referenced throughout `daypart_scheduler.py`, `dialogs.py`, `serialization.py`, and tests. Schedule this as a separate dedicated effort. After completing 3.1 (edit_tag accepting Tag), this refactor will be simpler.
