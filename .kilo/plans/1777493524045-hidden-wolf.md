**Difference:** The condition differs slightly:
- Anchor: `re.start_minutes < slot_end and re.end_minutes > slot_start`
- Fallback: `re.start_minutes < slot_end and re.end_minutes > current_pos`

However, note that `slot_start` equals the tag start (either anchored at `best_rand.end_minutes` or at `custom_start`), and at the point these loops run, `current_pos` has just been set to the tag's actual end via `_place_tag_videos`. Since the tag slot occupies `[slot_start, slot_end]` and `current_pos == slot_end` after placement, both conditions effectively check for entries overlapping the slot (anchor check uses slot boundaries, fallback checks if entry extends past `current_pos` which equals `slot_end`). They are semantically equivalent given the control flow.

## Appendix: Candidate Test Algorithms for Future Validation

The `ScheduleGenerator.apply_approximate(mode: str)` dispatch supports pluggable strategies. Below are candidate algorithms suitable for testing, comparison, and validation of the approximate scheduling system.

All strategies receive a `ScheduleGenerator` reference, granting access to:
- `self.tag_manager` — for tag lists and cache
- `self._place_tag_videos(...)`, `self._process_multi_series_tag(...)`, `self._process_series_tag(...)`, `self._process_custom_tag(...)`
- `_build_random_entries(...)`, `_create_video_entry(...)`

### A. Early-Fill (Earliest-Deadline-First)

**Mode name:** `"early_fill"`

**Logic:**
1. Sort all custom/series/multi-series tags by original `start_time` ascending (strictly earlier preferred start → earlier absolute placement)
2. For each tag on each day:
   - Compute its preferred absolute start: `orig_start + day_offset_minutes`
   - Place at `max(preferred_start, current_pos)` (no snapping to random boundaries)
   - Reserve full slot, call `_place_tag_videos` (or `_process_multi_series_tag`)
   - Update `current_pos = actual_end`
   - Consume overlapping random tails with `_consume_overlapping_tail(slot_start, slot_end, ...)`
3. Fill remaining gaps with random entries filtered through `occupied_ranges`/`scheduled_slots`

**Expected behavior:** Tags placed as close as possible to their preferred times without overlapping; current_pos only pushed forward by prior tags; no "best before" snapping.

**Test focus:** Verify no overlaps; tags maintain order; random fill correctly scheduled in gaps before first tag and after last tag.

---

### B. Late-Fill (As-Late-As-Possible)

**Mode name:** `"late_fill"`

**Logic:**
1. Sort tags by original `end_time` descending (later preferred end → later placement)
2. Process days independently; maintain `latest_free` pointer from day end backward
3. For each tag:
   - Compute its preferred absolute end: `orig_end + day_offset_minutes`
   - Backward placement: `slot_end = min(preferred_end, latest_free)`
   - `slot_start = slot_end - (orig_end - orig_start)`
   - Call placement helpers with `(slot_start, slot_end)`
   - Update `latest_free = slot_start`
   - Consumer tails from overlapping randoms still applied forward (may need variant logic)

**Expected behavior:** Tags pushed to the right, random fill crowded to the left of the day. Potential for early-day gaps if tags exceed day capacity.

**Test focus:** Day-end overflow handling; no overlaps; tail consumption still correct (though oriented backward).

---

### C. Priority-Queue / Weighted Anchoring

**Mode name:** `"priority"`

**Logic:**
1. Introduce `tag.priority` (int, default 0). Higher = more important.
2. Process tags in descending priority order; break ties by original `start_time`.
3. Important tags get first pick of anchor candidates; lower-priority tags get whatever space remains.
4. Option: allow lower-priority tags to be truncated or skipped if no viable slot.

**Expected behavior:** High-priority tags achieve tighter anchoring; low-priority tags show more displacement.

**Test focus:** Priority ordering respected; low-priority tags don't cause overlaps; `scheduled_slots` correctly blocks high-priority slots from random intrusion.

---

### D. Greedy Best-Fit Anchor

**Mode name:** `"best_fit"`

**Logic:**
- Current find-replace picks the candidate that ends closest to `custom_start` (minimum gap forward).
- This variant scores candidates by total wasted space: `(anchor_end - custom_start)` + `next_gap_after_tag`.
- Compute for each candidate the resulting free gaps before and after the tag slot; pick candidate minimizing sum of those gaps.
- Equivalent to "best-fit decreasing" in bin packing.

**Expected behavior:** Tighter packing; potentially fewer small leftover gaps that could cause random entries to slip between tags.

**Test focus:** Compare gap distribution against find-replace; ensure tail-consumption still captures all intruding randoms.

---

### E. Round-Robin Interleave

**Mode name:** `"round_robin"`

**Logic:**
1. Merge random fill segments and tag blocks into a single timeline first (ignoring conflicts).
2. Walk the day timeline alternately: schedule next random segment, then next tag block, then next random segment, etc.
3. When a tag's preferred `start_time` falls inside a random segment, truncate that random segment at the tag start and re-queue its tail later.
4. This interleaves tags and random fill at fine granularity rather than "tags-batch-then-fill."

**Expected behavior:** More mixed appearance of tags and random entries; may produce more realistic "flow"; `current_pos` updated after each element.

**Test focus:** Order preservation; no gaps/overlaps; tail-queuing correctness.

---

### F. Day-Spanning Enabled

**Mode name:** `"linear_spanning"` (modifier to Linear)

**Logic:**
- In current linear mode, tags are day-local: a tag starting at 23:00 with 2h duration is truncated at midnight.
- This variant allows tags to span across day boundaries: compute `slot_end`; if it exceeds `day_end`, continue placement into next day's timeline, adjusting `current_pos` across days and carrying `next_custom_pos` accordingly.

**Expected behavior:** Multi-series tags or long movies automatically wrap to day+1; no artificial truncation at midnight.

**Test focus:** Day-transition continuity; next day's tags shifted correctly; no overlaps across day boundary.

---

### G. Exhaustive Oracle (Small Instances)

**Mode name:** `"exhaustive"`

**Logic:**
- For ≤ `N` tags (e.g., N=4) and ≤ 2 days, enumerate all permutations of tag orderings and for each, simulate exact placement (linear or find-replace style).
- Evaluate each full schedule for:
  - Total overlap violations
  - Total tag displacement from preferred start
  - Random fill fragmentation
- Return the optimal schedule (minimal weighted score).

**Purpose:** Golden reference for small instances; used to assert that heuristic algorithms produce feasible (if suboptimal) schedules.

**Implementation note:** Should be guarded behind a flag; only enabled for small N due to factorial complexity.

```python
class MyNewStrategy:
    def __init__(self, generator: ScheduleGenerator):
        self.sg = generator
    def generate(self, num_days: int = 1) -> List[ScheduleEntry]:
        # ... implementation ...
        return final
```

2. Add dispatch branch in `ScheduleGenerator.apply_approximate`:
```python
def apply_approximate(self, num_days: int = 1, mode: str = "find_replace") -> List[ScheduleEntry]:
    if mode == "linear":
        return LinearApproximateStrategy(self).generate(num_days)
    elif mode == "find_replace":
        return FindReplaceApproximateStrategy(self).generate(num_days)
    elif mode == "my_new_mode":
        return MyNewStrategy(self).generate(num_days)
    else:
        raise ValueError(f"Unknown approximate mode: {mode}")
```
