## 1. Overview

The approximate scheduling feature (find-replace and linear modes) has critical bugs that cause **random fill entries to overlap custom/series tag time slots**. The user's 30-day calendar preview shows random entries starting inside series blocks (e.g., "Movie collection 01 - The Dark Legacy.mp4" at 13:19 inside Arcade hunters 13:00–15:15).

This plan documents the root causes and outlines concrete fixes.

---

## 2. Bug Details

### 2.1 Find‑Replace Mode — Overlap due to Partial Slot Occupation

**File:** `models.py`  
**Method:** `ScheduleGenerator._apply_approximate_find_replace`

**Symptoms:**
- Random entries appear at the same start time as custom/series episodes.
- Overlap occurs when a custom tag's episodes do not fully occupy its reserved time slot (i.e., `video_count` limited, total episode duration < `orig_end - orig_start`).

**Root Cause:**
1. After placing a tag, `current_pos` is set to the **end of the placed episodes**, not the **end of the tag's reserved slot**.
2. The gap between `current_pos` and the true slot end remains unmarked as occupied.
3. A random entry that overlaps this gap can have its "remaining portion" added starting at `current_pos` (inside the slot), or if it ends before `current_pos`, it may be added as a full entry because the overlap check compares only against **episode intervals**, not the full slot.
4. The `occupied_ranges` list is built exclusively from `final` entries (the episodes), so it does not include the unplayed gap.

**Concrete example from user:**
- Arcade hunters: reserved slot 13:00–15:15 (135 min), `video_count=4`. Episodes sum to ~135 min (exact), but even if they fill exactly, the algorithm conceptually leaves the gap unoccupied because episodes are separate intervals. Overlap check uses those intervals; an entry starting at 13:19 and ending at 15:31 overlaps the first episode (13:19–13:57) and should be rejected, but it slips through because the occupied check evaluates per-interval and might be bypassed due to ordering/gap logic.

**Fix Strategy:**
- Maintain a separate list `scheduled_slots` for the **full reserved intervals** of each tag as placed (after anchor/snapping).
- After all tags are processed for the day, extend `occupied_ranges` with `scheduled_slots` so the second-pass overlap check sees the entire tag block as occupied, including any internal gaps between episodes.
- Set `current_pos` to the **slot end** (`slot_end`) immediately after placing the tag, not to the episode chain end. This prevents tails from being inserted in the middle of the slot.
- Adjust the "remaining portion" logic: it should only add tails of random entries that extend past `slot_end`, starting at `slot_end` (or the later of `slot_end` and `current_pos` if tails already consumed some space). Simpler: after consuming the tail, update `current_pos` to `slot_end` to close the gap.

### 2.2 Linear Mode — Broken Truncation & Cross-Day State Leak

**File:** `models.py`  
**Method:** `ScheduleGenerator._apply_approximate_linear`

**Symptoms:**
- Random fill is appended in full without truncation, overlapping custom tags.
- `current_pos` and `next_custom_pos` are not reset per day, causing tags to be pushed forward on subsequent days.
- The `occupied` set is populated but never used to filter random entries.

**Root Cause:**
1. For non‑24h fill, the code builds `base_entries` (random fill for one day) and then simply appends them to `final` after custom tags without checking for overlap.
2. `current_pos` is maintained across days in the `has_24h_fill` branch, but even in the non‑24h branch, the logic that skips base entries based on `current_pos` is missing; instead, a dead `rand_idx` loop exists.
3. Series and custom tags both update `next_custom_pos` and `current_pos` cumulatively, causing day‑to‑day drift.

**Fix Strategy:**
- Rewrite the non‑24h path to mirror the find-replace approach: build a day‑local `occupied` set from scheduled tag slots, then filter/truncate random entries to fit gaps.
- Reset `current_pos` and `next_custom_pos` to `day_start` at the beginning of each day iteration.
- Use the same `scheduled_slots` pattern to mark full tag durations as occupied.
- Ensure `base_entries` are only used as a source of random videos, not appended blindly; instead, call `_build_random_entries` with the day's free intervals.

### 2.3 Series & Custom Video Ordering Ignored

**File:** `models.py`  
**Method:** `ScheduleGenerator._place_tag_videos` and `_process_custom_tag`

**Symptoms:**
- Series tags with `play_mode = "sequence"` episodes play in random order instead of episode sequence.
- Custom tags with `randomize_videos = False` also appear in random order.

**Root Cause:**
1. `_place_tag_videos` (used by find-replace for both custom and series tags) always shuffles the video list before placement:
   ```python
   videos = ct.collection_videos.copy()
   random.shuffle(videos)   # unconditional
   ```
2. For series tags, find-replace bypasses `_process_series_tag` (which correctly uses `parse_videos_for_series` to order episodes) and calls `_place_tag_videos` directly with the raw collection.
3. `_process_custom_tag` (used by normal mode and linear mode) also unconditionally shuffles, ignoring the `randomize_videos` flag.

**Fix Strategy:**
- Modify `_place_tag_videos` to accept an optional pre‑ordered video list (`videos_override`). If provided, use it directly; otherwise, copy from tag and shuffle **only if** the tag requests random order.
- For series tags in find-replace: compute ordered episodes via `parse_videos_for_series` and pass them as override.
- For custom tags: shuffle only when `ct.randomize_videos` is `True`.
- Update `_process_custom_tag` to conditionally shuffle based on `ct.randomize_videos`.

- New tag type (or composite) that groups multiple series tags.

  - New class `MultiSeriesTag` (or extend `Tag` with `is_multi_series` flag and `series_list: List[dict]`).

  - Integration with `ScheduleGenerator`: treat `MultiSeriesTag` similarly to series tags but expand internally into multiple series blocks.
  - Support both approximate modes: the multi‑series block should be reserved as one contiguous slot (sum of all parts) unless split across days; handle day boundaries by wrapping to next day if needed.

## 6. Risks & Notes

- The find‑replace fix changes `current_pos` semantics; ensure that subsequent tags still snap correctly.
- The `scheduled_slots` approach must handle multiple tags on the same day and correctly compute free space.
- Be cautious with identity checks (`re is rand_e`); if `_build_random_entries` recreates objects each call, the `is` check may fail. Currently `random_entries` is built once per generate call, so identity holds.
- Debug prints are useful during development; remove them only after fix is stable.
